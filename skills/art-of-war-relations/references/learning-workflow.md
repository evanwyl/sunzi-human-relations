# Learning workflow

这个 skill 支持“人物案例库”机制，让同一个人的互动可以逐步沉淀成可复用判断。

目标不是偷存隐私，而是把重复出现的互动模式整理成一个可成长的 case file。

## 核心原则

- 默认 **自动读，不自动写**
- 只有用户明确触发时，才把某次互动沉淀进人物案例库
- 每个人一份 case file
- 下次再次分析这个人时，先读 case file，再结合《孙子兵法》与《三十六计》判断
- case file 是 skill 自带能力的一部分，不依赖 OpenClaw memory 才能工作，便于开源复用
- 推荐配合 `cases/README.md`、`cases/indexes/people-index.md`、`cases/indexes/relationship-types.md` 一起使用

## 什么时候读

当用户的问题明显指向“同一个人”时，优先尝试命中已有人物案例库。

典型信号：
- 还是那个同事
- 继续分析小王
- 前任又来找我了，就是上次那个
- 这个人又来这一套了

如果能明确命中人物档案，先读取再回答。

## 什么时候写

只有在用户明确表达想沉淀时才写入。

推荐触发词：
- 沉淀这个人
- 记进案例库
- 把这段存进这个人的档案
- 以后这个人按这个思路分析

没有明确触发，不自动写。

## 人物目录建议

推荐目录：
- `cases/people/`：放具体人物档案
- `cases/indexes/people-index.md`：放人物索引
- `cases/indexes/relationship-types.md`：放关系类型约定

这样更适合开源仓库维护，也方便后续扩展。

## 人物标识建议

建议使用：
- 关系类型 + 名字/代号 + slug

例子：
- `coworker-xiaowang.md`
- `friend-ahui.md`
- `ambiguous-chenchen.md`
- `ex-lulu.md`

原因：
- 比纯名字更稳
- 比内部 ID 更可读
- 适合开源用户自己维护

## 每次沉淀该写什么

不是把整段对话原样塞进去，而是提炼成可复用结构：

- 这是谁
- 当前关系背景
- 本次发生了什么
- 对方呈现出的典型模式
- 用户当时的目标是什么
- 这次推荐了什么应对
- 哪个判断依据最关键
- 哪些说法或动作可能踩雷
- 下次遇到这个人优先检查什么

## 使用流程

### 流程 A：普通分析
1. 识别是否命中人物案例库
2. 如果命中，先读 case file
3. 结合当前问题 + 历史模式输出判断
4. 默认不写回

### 流程 B：用户要求沉淀
1. 完成当前分析
2. 把本次互动提炼成结构化记录
3. 更新对应人物 case file
4. 后续再遇到此人时优先读取

## 设计边界

- 不鼓励把普通关系全部阴谋化
- 不把人物档案写成情绪宣泄本
- 不靠一次互动就下永久定性
- 新记录可以修正旧判断，不把第一次判断当成绝对真相
- 人物案例库的价值在于识别重复模式，不在于放大偏见
